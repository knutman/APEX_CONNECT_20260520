Spatial & JSON — Bürger erster Klasse

SDO_GEOMETRY

  • Native Geo-Spalte mit SRID 4326
  • Spatial-Index spatial_index_v2
  • SDO_FILTER, SDO_WITHIN_DISTANCE & Co.
  • Voraussetzung für Vector Tiles

JSON

  • json-Spalte direkt im DDL
  • JSON_TABLE entfaltet Arrays im SELECT
  • Nested Paths für mehrstufige Strukturen
  • Kein PL/SQL-Parsing nötig

Das Problem

Was passiert?

  • APEX lädt die komplette Query
  • Jeder Pin landet im DOM
  • Initial-Payload: mehrere MB JSON
  • Render-Stutter bei jedem Pan/Zoom
  • Auf Tablet/schwacher Hardware: unbenutzbar

Hilft das?

  • Clustering im Frontend → lindert
  • Initial-Zoom kleiner → lädt trotzdem alles
  • Filter via Page-Item → ändert die Story

Wir brauchen eine andere Architektur.

Knut Göttling · Enignum AI

Was sind Kacheln?

Die Karte ist in quadratische Kacheln aufgeteilt. Jede Zoomstufe verdoppelt die Anzahl in jeder Richtung.

Z=0
1 Kachel
Z=1
4 Kacheln
Z=2
16 Kacheln
Z=3
64 Kacheln
  • Jede Kachel ist identifiziert durch (Z, X, Y) — Zoom, Spalte, Zeile
  • Browser lädt nur die sichtbaren Kacheln und cached sie wie Bilder
  • Bei Zoom n existieren 4n Kacheln
Knut Göttling · Enignum AI

Vector Tiles

  • Kachel-Inhalt im kompakten Binärformat (MVT, Mapbox Vector Tile)
  • Enthält Geometrien + Properties, kein vorgefertigtes Pixel-Bild
  • Symbol & Farbe legt das Frontend fest — vollständig stylebar
  • HTTP-cachebar, zweite Anfrage = 0 ms
  • Pro Kachel typisch wenige KB statt MB-JSON

Vector Tiles aus Oracle

Eine Funktion. Eine Tabelle. Ein BLOB.

function get_tile(p_zoom number, p_x number, p_y number) return blob is
  l_tile blob;
begin
  l_tile := MDSYS.SDO_UTIL.GET_VECTORTILE(
    table_name    => 'CHA_CHARGING_STATION',
    geom_col_name => 'GEO',
    tile_x        => p_x,
    tile_y        => p_y,
    tile_zoom     => p_zoom,
    att_col_names => MDSYS.SDO_STRING_ARRAY(
                       'CHA_CHARGING_STATION_ID',
                       'STATION_TYPE',
                       'OPERATOR_DISPLAYNAME'),
    layer_name    => 'charging_stations',
    google_ts     => true            -- WICHTIG: XYZ-Schema
  );
  return l_tile;
end;

ORDS-Endpoint — in 30 Zeilen

ords.define_module(
  p_module_name => 'cha',
  p_base_path   => 'cha/');

ords.define_template(
  p_module_name => 'cha',
  p_pattern     => 'tiles/:z/:x/:y');

ords.define_handler(
  p_module_name => 'cha',
  p_pattern     => 'tiles/:z/:x/:y',
  p_method      => 'GET',
  p_source_type => ords.source_type_plsql,
  p_source      => q'[
    declare l_tile blob;
    begin
      l_tile := PA_TILES.get_tile(:z, :x, :y);
      owa_util.mime_header('application/vnd.mapbox-vector-tile', false);
      htp.p('Cache-Control: public, max-age=3600');
      owa_util.http_header_close;
      wpg_docload.download_file(l_tile);
    end;
  ]');

Take-aways

  1. Moderne Oracle-Features machen das Datenmodell selbstbeschreibend.
  2. Spatial & JSON sind Bürger erster Klasse — kein Add-on mehr.
  3. Vector Tiles aus der DB lösen das Skalierungsproblem klassischer Pin-Karten.
