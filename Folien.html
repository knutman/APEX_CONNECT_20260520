Der Datensatz

  • Frei verfügbar als REST-Endpoint, JSON

Spatial & JSON — Bürger erster Klasse

SDO_GEOMETRY

  • Native Geo-Spalte mit SRID 4326
  • Spatial-Index spatial_index_v2
  • SDO_FILTER, SDO_WITHIN_DISTANCE & Co.
  • Voraussetzung für Vector Tiles

JSON

  • json-Spalte direkt im DDL
  • JSON_TABLE entfaltet Arrays in SELECTs
  • Nested Paths für mehrstufige Strukturen
  • Kein PL/SQL-Parsing nötig

Loader: REST direkt im MERGE

Keine Staging-Tabelle, keine Zwischenstufe.

merge into cha_charging_station dst
  using ( select res.*
            from json_table(
                   APEX_WEB_SERVICE.make_rest_request(
                     p_url => lcUri, p_http_method => 'GET'),
                   '$.chargingStations[*]'
                   columns (
                     ext_id   number   path '$.id',
                     city     varchar2 path '$.city',
                     evses    clob format json path '$.evses',
                     ...
                   )
                 ) res
        ) src
  on            (dst.ext_id = src.ext_id)
  when matched then update set ...
  when not matched then insert  ...;

Das Problem

Was passiert?

  • APEX lädt die komplette Query
  • Jeder Pin landet im DOM
  • Initial-Payload: mehrere MB JSON
  • Render-Stutter bei jedem Pan/Zoom
  • Mobil / schwache Hardware: unbenutzbar

Hilft das?

  • Clustering im Frontend → linderts
  • Initial-Zoom kleiner setzen → lädt trotzdem alles
  • Filter via Item → ändert aber die Story

Brauchen wir eine andere Architektur.

Vector Tiles

  • Karte ist in Kacheln aufgeteilt: Z (Zoom), X, Y
  • Pro Kachel ein kompaktes Binärformat (MVT, Mapbox Vector Tile)
  • Browser lädt nur die sichtbaren Kacheln
  • Cache-fähig wie ein Bild — zweite Anfrage = 0 ms
  • Symbol & Farbe werden im Frontend festgelegt

Vector Tiles aus Oracle

Eine Funktion. Eine Tabelle. Ein BLOB.

function get_tile(p_zoom number, p_x number, p_y number) return blob is
  l_tile blob;
begin
  l_tile := MDSYS.SDO_UTIL.GET_VECTORTILE(
    table_name    => 'CHA_CHARGING_STATION',
    geom_col_name => 'GEO',
    tile_x        => p_x,
    tile_y        => p_y,
    tile_zoom     => p_zoom,
    att_col_names => MDSYS.SDO_STRING_ARRAY(
                       'CHA_CHARGING_STATION_ID',
                       'STATION_TYPE',
                       'OPERATOR_DISPLAYNAME'),
    layer_name    => 'charging_stations',
    google_ts     => true          -- WICHTIG: XYZ-Schema
  );
  return l_tile;
end;

ORDS — REST-Endpoint in 30 Zeilen

ords.define_module(
  p_module_name => 'cha',
  p_base_path   => 'cha/');

ords.define_template(
  p_module_name => 'cha',
  p_pattern     => 'tiles/:z/:x/:y');

ords.define_handler(
  p_module_name => 'cha',
  p_pattern     => 'tiles/:z/:x/:y',
  p_method      => 'GET',
  p_source_type => ords.source_type_plsql,
  p_source      => q'[
    declare l_tile blob;
    begin
      l_tile := PA_TILES.get_tile(:z, :x, :y);
      owa_util.mime_header('application/vnd.mapbox-vector-tile', false);
      htp.p('Cache-Control: public, max-age=3600');
      owa_util.http_header_close;
      wpg_docload.download_file(l_tile);
    end;
  ]');

Aspekt App 1 Pins App 2 Vector Tiles
Initialer Payload mehrere MB JSON ~KB pro Tile
Renderzeit sekundenlanger Stuttersofort
Pan / Zoom alle Pins neu nur fehlende Tiles

Aufwand Frontend APEX-Standard MapLibre-Glue
Lohnt sich ab ~5.000 Features

Take-aways

  1. Moderne Oracle-Features machen das Datenmodell selbstbeschreibend.
  2. Spatial + JSON sind Bürger erster Klasse — kein Add-on mehr.
  3. Vector Tiles aus der DB lösen das Skalierungsproblem klassischer Pin-Karten.
